Systems case study — recommendation infrastructure

Ranking a catalog in 16 numbers, in under 150 milliseconds

How a vector-search recommendation engine added real, per-user ranking on top of an eligibility-only rules engine — and stayed up 99.99% of the time while doing it, at a scale of tens of millions of requests a day.

01 — The problem

Same offer, every user, every time

Before this system existed, the recommendation surfaces inside the product — the banners and cards nudging a user toward a relevant feature or upgrade — were driven by a rules engine. Rules are fine for "don't show this to ineligible users." They're bad at "which of these forty eligible offers is actually relevant to this person, right now." A rules engine can exclude; it can't rank.

Rules are good at eligibility — which segment can see which offer, which combinations are excluded, which compliance constraints apply. What they can't do is rank. A rules engine can narrow forty eligible offers down to the five a user qualifies for, but it has no way to know which of those five is actually the most relevant to that specific person, based on what they've actually been doing in the product.

The goal was never to throw out the rules engine — eligibility and compliance logic still need to be deterministic, and you don't want a ranking model quietly learning to violate a hard exclusion rule. The goal was to put a real ranking layer on top of it: something that scores relevance per user, using their behavior, instead of stopping at eligible-or-not. Understanding Vespa's scoring model well enough to design the schema and rank profiles around it, embedding the content-side vectors, and building the pipeline that keeps those vectors correct as content changes — that freshness problem ended up being the part that ate the most time.

02 — Why Vespa, specifically

The filtering requirement ruled out a plain vector database

The natural first instinct for "rank things by relevance" is a managed vector database — Pinecone, Weaviate, something you can stand up in an afternoon. We evaluated that path and moved off it fairly quickly, for a reason that's easy to miss until you hit it in production: ranking and filtering aren't separate steps here, they have to happen in the same request, on the same infrastructure. A recommendation has to be filtered by which product surface it's eligible for, which items the user has already seen, and business rules that can't leak into the score — before it ever gets ranked.

vs. a managed vector DB (Pinecone-style)

Great at pure nearest-neighbor search, but metadata filtering is bolted on and there's no way to express a custom ranking expression that blends a dot-product with business logic. We'd have needed to build an orchestration layer on top just to combine "filter by eligible surface" with "rank by affinity" — which defeats the point of using a managed service.

Ruled out: filtering + custom ranking can't both be first-class

vs. Elasticsearch's kNN

ES added vector search, but it's a retrofit on a text-search engine. There's no native tensor type — you're working around the index rather than with it. Vespa, by contrast, lets you define an embedding as a first-class tensor field in the schema and write the ranking expression as actual math, evaluated where the data lives.

Ruled out: vectors are a bolt-on, not a native concept

vs. Weaviate

Weaviate does hybrid search well, but the "searcher chain" pattern — load the user's vector, apply a structured filter, then rank, all inside one request on one node — isn't a first-class concept the way it is in Vespa. You'd be shipping that orchestration back and forth over the network, which is exactly the latency tax we couldn't afford.

Ruled out: no native concept of chained, in-process search stages

What made Vespa the right call wasn't any single feature — it was that filtering, ranking, and serving all happen in the same process, on the same node, with no network hop in between. At the request volume we were running, that architectural difference is the difference between a 150ms system and a 500ms one.

03 — Architecture

Two separate concerns: keeping content fresh, and serving a ranked list fast

The system splits cleanly into two paths that almost never touch each other: an ingestion path that keeps the content vectors up to date, and a serving path that answers a recommendation request. Keeping these physically separate — down to running on different nodes — was deliberate: a slow or bursty content update should never be able to degrade a live user-facing request.

ingestion never shares resources with serving — a content backlog can't touch p99

On the ingestion side, content changes don't get pushed straight into Vespa. They land in SQS first, which exists purely so the systems publishing content changes don't need to know or care about our processing capacity — SQS absorbs bursts and gives us dead-letter handling for free. From there, a Lambda function bridges SQS to several downstream Kafka topics, fanning the same event stream out to the different services — ours included — that need it. Kafka is what actually gives us the property that matters most operationally: replay. If we ship a bad embedding model or find a bug in how tags are weighted, we can rewind the topic to an earlier offset and rebuild the index without touching the upstream systems at all.

From there, updates land in Vespa as partial updates, not full document replacements. That distinction matters more than it sounds — a partial update touches only the changed field in Vespa's in-memory attribute store, so there's no reindex lag and no window where a query could see a half-written document.

The serving side is a short, deliberately thin chain: one searcher attaches the user's embedding to the query, another applies the eligibility and "already-seen" filters, and then Vespa's own ranking runs natively on the content node — no Java, no network hop, just the dot-product math running where the data already lives. That's the part that makes 150ms achievable at this volume: the expensive computation never leaves the machine it's running on.

04 — Design decisions

Dot-product over cosine, and why 16 dimensions is a feature, not a limitation

Two choices get questioned in almost every technical conversation about this system, so they're worth explaining properly rather than glossing over.

Dot-product ranking, not cosine similarity

Cosine similarity normalizes away magnitude — it only cares about direction. That's the wrong model here. A niche item with a weak-but-aligned signal shouldn't rank the same as a broadly strong one; magnitude is meaningful because it encodes confidence, not just direction. The ranking expression is a straightforward sum(query(user_vector) × attribute(item_vector)) — evaluated per document, in-memory, with no normalization step in the way.

16-dimensional embeddings

This looks small next to a 128 or 512-dimension text embedding, until you remember this isn't open-domain semantic search — it's matching users against a bounded catalog with well-understood latent factors (category, behavioral affinity, product context). Those larger dimensions exist to capture an effectively unbounded concept space; ours isn't unbounded. The practical payoff is real: a 16-float vector is 64 bytes, small enough that the entire working set of embeddings sits comfortably in a content node's cache. At the request volumes involved, staying in cache rather than falling back to memory bandwidth is often the actual difference between a fast system and a slow one.

The filtering and custom-ranking requirements ruled out the simpler paths from day one. This wasn't over-engineering — it was the minimum architecture the requirement actually needed.
05 — Latency engineering

Six things had to be true simultaneously to hit p99

No single optimization gets you to 150ms at this scale — it's the compounding effect of removing every avoidable millisecond from the request path.

  • Thin searcher chain. Each Java-layer searcher does the minimum possible work — attach a vector, build a filter — and hands off immediately. The expensive part runs natively on the content node, not in the request-handling layer.
  • Filter before rank, not during. Items excluded by eligibility or "already seen" logic are dropped before they ever enter the ranking pipeline, not scored and then discarded. Skipping the ranking pipeline entirely for filtered-out terms avoids a meaningful chunk of wasted computation at this volume.
  • In-memory attribute fields. The fields that matter for ranking — ids, tags, the embedding tensor itself — are held in memory, not on disk. The entire scoring computation happens with zero disk I/O in the hot path.
  • Compact tensors. A 64-byte embedding isn't just "small" in the abstract — it's small enough to survive in L2/L3 cache across the whole working set, which is the difference between a cache hit and a cache miss on every single ranking call.
  • Physical isolation of feed and serve. Document ingestion runs on a separate container, on a separate node, from the one answering user requests. A slow or bursty content update literally cannot compete for CPU or memory with a live request.
  • Per-segment monitoring, not just an aggregate number. Watching p99 as one blended number hides regressions — a degrading path on one product surface or device type gets averaged away by a healthy one elsewhere. Breaking latency down by segment is what actually lets you catch a regression before a customer does.
06 — Reliability

99.99% isn't one mechanism, it's five, each covering a different failure

Every "nine" in an availability target has to be backed by an actual mechanism, or it's just a number on a slide. Here's what actually covers each failure mode:

Region failure

Multi-region active-active deployment with automatic failover — if one region degrades, traffic shifts without a human in the loop.

Bad deployment

A canary instance takes every change first, with a bake window before it reaches the second region — so a bad rollout is contained to a fraction of traffic, not all of it.

Node failure

Content redundancy means every document lives on more than one node — a single node going down doesn't lose data or drop queries.

Cascading failure

Feed and serve run on physically separate containers, so a problem in document ingestion can't cascade into a serving outage.

Dependency failure

If the external user-profile store is unreachable, the system falls back to a default profile rather than failing the request outright — degraded personalization beats no response.

07 — What I'd own confidently in an interview

The gaps, and what comes next

Every system has known tradeoffs, and being specific about them tends to land better than pretending the system is finished.

User embeddings are fetched synchronously at query time

This is the simplest correctness model — the vector is always current, no staleness risk. The cost is a live dependency on an external store sitting directly in the request path.

→ Next: pre-fetch and cache at session start with a short TTL, to take that store out of the critical path without sacrificing freshness.
No real-time collaborative filtering

Item embeddings are pre-computed offline; "users like you also liked" signal is baked in rather than computed live.

→ Next: stream-compute collaborative signal from the same event pipeline that already exists for content updates.
Model-quality degradation is inferred, not measured directly

We watch engagement metrics and ranking-score drift as proxies for "is the model still good," but a proxy is still a proxy.

→ Next: shadow-score a candidate model against production traffic and compare distributions before it's ever fully rolled out.
